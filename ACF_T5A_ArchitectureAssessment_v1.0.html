Executive Summary

This document provides a comprehensive architecture assessment of ACF's current technology estate and designs the future-state architecture required to support the 3-stage corporate strategy (Stabilize, Grow, Lead). The current architecture is characterized by monolithic core banking (T24 R25iACF IT Division, "20251115_BDIT v5.pptx" AdvisorsData), point-to-point integrations, no API management, no data platform, critical security gaps (no firewalliEY "Executive Summary Report_v.3.1.pptx" — Cybersecurity Findings, no SIEMiEY "Executive Summary Report_v.3.1.pptx" — Cybersecurity Findings, no WAFiEY "Executive Summary Report_v.3.1.pptx" — Cybersecurity Findings per EY findings), and single-site hosting with unvalidated DR. The future-state design transitions ACF to an API-first, event-driven, cloud-ready, composable architecture capable of supporting 800,000+ customers, 5+ product lines, real-time AI/ML decisioning, and embedded finance across the AEON ecosystem.

1. Current State Architecture IKIGAI DESIGN

1.1 Application Architecture Map IKIGAI DESIGN

ACF's application landscape consists of a fragmented set of systems with limited integration maturity. The following map describes the current production and in-development systems.

1.1.1 Core Banking Layer

**T24 Transact**
Platform: Temenos
Version: R25 (upgrade in progress)
Status: Active
Function: Core banking: accounts, lending (AA module), GL, FX, MM, billing, delinquency management, WAON point redemption, PTF point payment
**T24 Batch Upload**
Platform: Temenos
Version: R25
Status: Active
Function: Bulk data processing for end-of-day/month operations
**T24 Close of Business (COB)**
Platform: Temenos
Version: R25
Status: Active
Function: End-of-day batch processing; creates downtime windows for early settlement, partial prepayment

Key Findings:

  • T24 operates on IBM Power Systems with Oracle RAC 19c database
  • Cannot support automated 5-group R18-based loan classification (SBV regulatory requirement)
  • Cannot support back-up date or semi-auto processing simultaneously with normal monthly automation
  • Monthly closing requires multi-system orchestration: Credit Screening System -> Card Solution System -> Core Banking -> final closing result sharing
  • Significant customizations for ACF-specific products (cash loan, installment CD/TW, revolving, ACS cross-sell) create upgrade complexity
  • BPR documents finalized for Lending (v1.6), Account (Final), Customer (v1.3 Final), GL (v3.1 Final), FX (v2.1 Final), MM (Final), Batch Upload (v2.0)

1.2.2 Payment API Architecture (v1.2)

The Payment API is the most documented integration layer and serves as middleware between collection partners and backend systems.

Key APIs:

  1. getLoansInfo -- Query loan information by loan_id and partner_id
  2. getLoansInfoLegal -- Query loans by legal_id (CCCD number) and partner_id
  3. createFTLoans -- Cash disbursement via T24 Adapter
  4. createFTLoansToAcc -- Bank account disbursement via SeABankiACF Banking Partner Agreement — Disbursement Channel
  5. CreateFTPayment -- Scheduled repayment collection via T24

Database TaiACF Banking Partner Agreement — Disbursement Channelbles: PARTNER, LOANS_INFO, LOANS_PAYMENT_INFO Disbursement Status Tracking: READY -> PAID Multi-channel Disbursement Fallback: SeABank direct -> CITAD -> NAPAS fast
